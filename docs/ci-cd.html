Operations

CI / CD

Every pull request runs the same gate you run locally — lint, type and Svelte diagnostics, Vitest — plus static security analysis and a Lighthouse audit. Merging to main deploys the Worker to Cloudflare automatically.

Pipeline at a glance

Workflow File Triggers Purpose
CI .github/workflows/ci.yml pull_request, push: main Parallel lint, check, test; build verifies the Worker bundle.
Deploy .github/workflows/deploy.yml push: main, workflow_dispatch pnpm build then wrangler deploy against the production GitHub Environment.
CodeQL .github/workflows/codeql.yml pull_request, push: main, weekly cron JavaScript / TypeScript static analysis (build-mode: none) with the security-and-quality query suite.
Lighthouse .github/workflows/lighthouse.yml pull_request, workflow_dispatch Boots the built Worker via wrangler dev --local and runs Lighthouse CI against the canonical routes. Budgets in lighthouserc.json.
Dependabot .github/dependabot.yml weekly cron Grouped npm (pnpm-aware) and github-actions updates with ecosystem-aware groupings (SvelteKit, Tailwind, Cloudflare, lint, types).

Required secrets

The deploy workflow is the only path that needs Cloudflare credentials. Both secrets are provided at the GitHub organization level and inherited automatically by this repository — no repo-level configuration is required.

Secret Scope Notes
CLOUDFLARE_API_TOKEN Organization Custom token with Account → Workers Scripts: Edit, Account → Account Settings: Read, User → Memberships: Read. Scope to the production account only.
CLOUDFLARE_ACCOUNT_ID Organization The Cloudflare account ID that owns the memenow-site Worker.

Production environment

deploy.yml targets a GitHub Environment named production. The Worker serves both https://memenow.xyz (apex) and https://www.memenow.xyz; the Environment URL records the apex as the canonical deployment link. The Environment is used for GitHub Deployments tracking; no required reviewer is configured today, so the merge-to-main path deploys without manual approval. Add a required reviewer in Settings → Environments → production if a human gate becomes necessary.

concurrency: deploy-production with cancel-in-progress: false ensures consecutive deploys serialize rather than clobber each other.

Quality gates on every pull request

  1. Lint — prettier --check . then eslint ..
  2. Type / Svelte check — svelte-kit sync followed by svelte-check against tsconfig.json (strict + checkJs).
  3. Vitest — both projects (client browser mode and server Node), with the Playwright Chromium binary cached under ~/.cache/ms-playwright.
  4. Build — fans in from the three checks and runs pnpm build to confirm the Cloudflare Worker bundle assembles. Bundle size is logged to the run summary.
  5. CodeQL — JavaScript / TypeScript analysis with the security-and-quality suite.
  6. Lighthouse CI — three runs per URL, desktop preset, budgets defined as warn-level assertions until the baseline is established.

Caching

  • pnpm store — handled by actions/setup-node@v4 with cache: pnpm; the cache key follows pnpm-lock.yaml.
  • Playwright Chromium — explicit actions/cache@v4 over ~/.cache/ms-playwright, keyed by pnpm-lock.yaml so a lockfile change invalidates the binary.
  • SvelteKit build output — not cached. svelte-kit sync is cheap, and stale caches risk drifting against tsconfig.json references.

Lighthouse budgets

lighthouserc.json declares performance, accessibility, best-practices, and SEO assertions plus concrete numeric budgets: LCP < 2500 ms, CLS < 0.1, TBT < 200 ms, FCP < 1800 ms, total transfer < 1 MB. All assertions are warn-level for now so flaky CI runs do not block merges; promote to error after two to three calm weeks of green runs.

Manual triggers

The deploy and Lighthouse workflows accept workflow_dispatch. To deploy manually: 

gh workflow run deploy.yml --ref main
# or, with a dry run:
gh workflow run deploy.yml --ref main -f dry_run=true

The dry_run input swaps wrangler deploy for wrangler deploy --dry-run --outdir=.wrangler/dry-run, producing the bundle without publishing it.

Rollback

Roll back from the latest successful deploy via the Cloudflare dashboard or:

wrangler rollback --message "incident-2026-05-11"

Alternatively, revert the offending commit on main. The push will retrigger deploy.yml and publish the previous bundle.

Failure runbook

Symptom First action
ci.yml / lint failing Reproduce with pnpm lint; auto-fix with pnpm format when possible.
ci.yml / check failing Run pnpm exec svelte-kit sync && pnpm check locally; inspect the Svelte and TypeScript diagnostics.
ci.yml / test flake Download the vitest-failure-artifacts artifact from the run; replay with pnpm test --reporter=verbose.
deploy.yml fails on wrangler deploy Check the Cloudflare API token scope and quota; rerun with dry_run=true first; if the bundle exceeds Worker limits, profile with pnpm build locally.
Lighthouse warns repeatedly on a single page Open the temporary report URL in the action log; treat budget warnings as a recommendation, not a block.
CodeQL alert Review under Security → Code scanning; if it is a false positive, dismiss with a reason; if real, fix in a follow-up PR.

Local sanity checks before pushing

pnpm lint && pnpm check && pnpm test
# Optional, for Worker, hook, or binding changes:
pnpm preview

These mirror the CI gate exactly. The /verify Claude skill wraps the trio.

Pinning policy The deploy workflow holds the only credential-bearing action invocation (cloudflare/wrangler-action@v3). If supply-chain hardening becomes a requirement, replace the major tag with the published 40-character commit SHA. Dependabot will continue to surface major-version upgrade PRs in either case.
